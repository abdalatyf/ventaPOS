# VentaPOS NextGen Licensing Architecture

This document provides the definitive technical specification of the VentaPOS Licensing Subsystem as integrated into the NextGen codebase. It covers the cryptographic validation protocol, local database storage, and runtime middleware gates.

> [!IMPORTANT]
> The License Generator is an offline, standalone application maintained strictly by the developer. It is **never** bundled with the VentaPOS NextGen executable to prevent reverse engineering and unauthorized key generation.

---

## 1. Licensing Model Overview

VentaPOS maps client permissions to specific package identifiers (`product_id` 1 to 16). The packages are split into four main categories: Trial, Time-based plans, Cloud sync add-ons, and Balance recharge packages.

### Product ID Mapping Table

| :--- | :--- | :--- | :--- |
| **1** | `تجربة مجانية (Trial 60 Days)` | Trial | 60 Days (calculated dynamically on client) |
| **2** | `طباعة فقط - سنة واحدة` | Time-based Basic | 365 Days |
| **3** | `طباعة فقط - مدى الحياة` | Lifetime Basic | 99 Years (36,135 days) |
| **4** | `إدارة كاملة - سنة واحدة` | Time-based Pro | 365 Days |
| **5** | `إدارة كاملة - 3 سنوات` | Time-based Pro | 1095 Days |
| **6** | `إدارة كاملة - 5 سنوات` | Time-based Pro | 1825 Days |
| **7** | `إدارة كاملة - مدى الحياة (VIP)` | Lifetime Pro | 99 Years (36,135 days) |
| **8** | `إضافة الأونلاين - سنة` | Cloud Add-on | 365 Days |

| **16** | `كود الطوارئ` | System Unlock | Single-use Reset |

## 2. Cryptographic Protocol (80-Bit Code)

The licensing code generated by the developer is an 80-bit integer block encapsulating the machine fingerprint, start date, and product type, signed via HMAC-SHA256, masked via bitwise XOR operations, and formatted into custom Base32 (`XXXX-XXXX-XXXX-XXXX`).

### Bitwise Structural Layout
1. **Signature (`final_sig`)**: Upper 36 bits (bits 44 to 79).
2. **Masked Data (`masked_data`)**: Lower 44 bits (bits 0 to 43).
   - **Machine Hash**: 32 bits (bits 12 to 43).
   - **Date Code**: 8 bits (bits 4 to 11).
   - **Product ID**: 4 bits (bits 0 to 3).

### Client-Side Validation
When the user submits an activation key (`license_code`), the POS app (`LicenseValidator`):
1. **Replay Check**: Hashes the key and checks if it exists in the `UsedLicense` model.
2. **Unmasking**: Recalculates the XOR mask using the signature and extracts the machine hash, date, and product ID.
3. **HMAC Verification**: Rebuilds the payload using the hardcoded obfuscated secret key and verifies the HMAC signature.
4. **Hardware Binding**: Matches the extracted machine hash against the local device's hardware fingerprint (`machine_id`).

---

## 3. Database Schema (`ClientLicense`)

Once validated, the license is saved locally in the SQLite database:
- `product_id`: The package ID.
- `start_date` / `expiry_date`: Subscription boundaries.
- `invoices_balance`: Remaining invoice quota (for recharge cards).
- `license_code_hash`: A cryptographic signature of the row itself to prevent direct SQLite tampering.

---

## 4. Runtime Enforcement Middleware

The Django application uses a middleware layer to enforce license boundaries on every HTTP request.

### Forensic Verification Check
The middleware reads the active `ClientLicense`. It recalculates the expected `license_code_hash`. If it does not match (meaning the user altered the SQLite DB manually), the license is immediately flagged as `is_active=False`.

### Security Gates
1. **Time Gate**: Ensures at least one active base license is present and not expired.
2. **Balance Gate**: If the request is saving a receipt, it checks if the aggregated invoice balance is greater than 0.
3. **Cloud Gate**: If the client is a secondary terminal, verifies cloud sync license.
